Case study 03 · Agentic delivery

Built 5 PM agents around Ralph so my time goes to discovery and design

Delivery used to consume the PM hours I'd rather spend on discovery and design. I built 5 agents (discovery, prototyping, PRD, acceptance testing, orchestration) around Ralph, an open-source autonomous coding agent that handles implementation. The pipeline runs review-gated: agents do the artifact work; I make the product calls.

Why delivery was slow

Product delivery follows a predictable loop - discovery, design, implementation, testing - but it's slowed by context switching across tools and artifacts, handoff latency between phases, and manual verification and rework. Even straightforward features could take days from idea to merged PR. This pipeline is how I continue to accelerate the same problem I've been solving since day one - just at a higher leverage point.

What I built

Five PM agents I built wrapping Ralph, a third-party autonomous coding agent that handles implementation. Each stage owns one phase of delivery, running in sequence with clear human checkpoints so my time stays on discovery, design, and the review gate.

01 · Discovery

Searches Slack, Gmail, meeting transcripts, competitors, and the codebase for context. Frames the problem and produces a structured discovery doc - JTBD framing, competitive insights, solution options - with napkin-sketch wireframes to make options concrete.

02 · Prototyping

Converts discovery into UI prototypes built with real components from the design system, in the codebase. Follows predefined design rules and iterates in a loop, capturing screenshots of the prototype for review and improvement.

03 · PRD

Turns the reviewed prototype and decisions into a PRD broken into user stories with explicit acceptance criteria - the contract the build and test phases run against.

04 · Implementation - Ralph (open-source)

Ralph is an open-source autonomous coding agent. It reads the PRD's user stories, implements, runs checks, commits, and opens a PR. My contribution at this step is the PRD→prd.json converter that feeds Ralph and the orchestration around it; the code-generation horsepower is Ralph's.

05 · Testing & orchestration

Reads the PRD acceptance criteria, spins up a test DB, drives the browser via Playwright, and outputs a structured test report with screenshots and a product announcement. Orchestration wires the phases together so the pipeline runs end-to-end with review gates between stages.

How it works - review-gated

I give the discovery agent a feature idea in plain English. It produces a discovery doc; I review and decide. The prototyping agent produces prototypes; I review and write the PRD. Implementation opens a PR; the test agent validates against acceptance criteria. My role stays on product judgment and tradeoffs. The agents handle execution.

The shift I cared about is where my time goes. PRD writing, prototype iteration, and acceptance-test scripting used to be hands-on PM work. They now run through agents I review and approve, freeing me to spend more time in discovery and design. As a system-level signal, the combined pipeline (5 PM agents + Ralph) shipped at 2,374 lines/week post-adoption versus 1,281 lines/week prior, a +85.4% lift in throughput. The implementation horsepower is Ralph's; the PM scaffolding I built is what makes the output reviewable and on-spec at the gate.

What I learned

  • The PM bottleneck isn't skill - it's the information-gathering before every decision. Agents that handle discovery, prototyping, and testing free judgment to stay where it matters: the review gate.
  • Acceptance criteria are the contract. When the PRD names what "done" means, Ralph can implement and the test agent can verify deterministically. Confidence at merge goes up, not down.
  • Integrate frontier agents, don't reinvent them. Ralph already solved code generation. My leverage was in the scaffolding around it: agents that produce the PRDs Ralph implements against, and the ones that test what it produces.
  • Don't just ship features - build the machine that ships them. This is the operate → build → automate pattern applied to delivery itself.
