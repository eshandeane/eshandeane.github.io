Case study 04 · AI in the product

5 AI features shipped into the onboarding platform

Not just using AI to build faster - building AI into the product for end users. Every feature came from a specific problem I observed during 34 manual onboardings, and every AI output keeps a human control: accept, edit, or dismiss.

The throughline

The time lost searching transcripts, the hours writing specs and tickets, the context that died in meeting recordings - these shaped what I built and why. All five features are covered by an LLM-as-Judge eval pipeline in Langfuse, with Langfuse telemetry for production monitoring.

1 · Ask FDE - RAG-powered workspace Q&A

Onboarding analysts run 5–6 back-to-back calls daily across 10–17 concurrent accounts. Between calls, answering "what's blocking this account right now?" used to mean switching between Circleback transcripts, the spec document, and Jira. Ask FDE is a conversational assistant embedded in each workspace, grounded in three sources - the specification, uploaded meeting transcripts, and synced Jira tickets - using a per-workspace semantic search store. Every answer carries source citations so users see exactly which meeting, spec section, or ticket it came from, and it degrades gracefully to spec-only context before meetings are uploaded.

2 · Meeting-to-account matching

Every call generates decisions and commitments that die in recordings unless someone extracts and connects them. A webhook integration ingests recordings from Circleback and Fathom, and an AI step identifies the correct distributor account by analyzing attendees, title, and notes - no manual assignment. Approved meetings are indexed into per-workspace search stores, forming the data backbone for Ask FDE and the Control Tower. (287 historical recordings were sitting unmatched before this shipped.)

3 · Spec generation

Once questionnaire responses are collected across four intake stages, someone has to read all of it and write a structured specification across seven categories - the source of truth for the entire integration. One-click generation reads all completed responses and produces a fully editable spec, run in parallel per section. Delta reviews and corrects before it goes to the distributor for sign-off, and the signed spec feeds ticket generation downstream.

4 · Jira ticket generation

After sign-off, a three-phase AI pipeline parses the spec into categories, generates tickets per category using domain-specific prompts, and persists them as reviewable drafts. Managers approve, edit, or reject individual tickets before a batch export to Jira with parallel execution and relevant data files attached - replacing the manual creation of 30–80 tickets per account.

5 · Weekly update drafting - built and shipped solo, end-to-end

This one I owned entirely: spec, prompt design, evals, UI, and deployed code. Writing weekly updates across 10–17 accounts was a top pain point - one senior manager did it on weekends because there was no time during the week. Now an AI draft is generated automatically when a meeting is approved, answering the team's standard five-question format and grounded in the transcript. Three affordances: post as-is, edit, or dismiss. An earlier version over-dramatized issues ("AI will pick up one thing and call it a critical blocker"), so I redesigned the prompts around evidence grounding, not inference.

The eval discipline

The AI PM work isn't shipping AI - it's knowing when it's wrong. I designed an LLM-as-Judge eval pipeline across all five features and used it to surface that the RAG assistant's grounding averaged 46.7%. The eval broke the number down: open-ended summary prompts were scoring 0–40%, while specific questions scored 95%+. That diagnosis pointed straight at the fix - I prioritized a citation prompt change rather than a broad model swap.

Why this matters for an AI PM role

An average grounding number looks like a model problem. The eval pipeline turned it into a prompt problem with a known cause and a cheap fix. That's the difference between speccing AI features and owning them.
